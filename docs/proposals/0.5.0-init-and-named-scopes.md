# Propuesta de cambios: Tracevault 0.5.0 — `initTracevault` + scopes por alias

> **Obsoleta — sustituida por la propuesta final:** [`unified-app-instance-bootstrap.md`](./unified-app-instance-bootstrap.md) (**Tracevault 1.0.0**: `startTracevault`, dos pools read/write, `getScope`). Este archivo se conserva solo como referencia histórica del diseño “init solo DDL + fábricas separadas”.

---

## 1. Objetivos

1. **`initTracevault(options)`** — Únicamente **provisionar la base de datos**: crear/asegurar tablas e índices necesarios para cada `tableName` declarado en el mapa de scopes, de forma **idempotente**. No sustituye `createTracevault` ni `createTracevaultQuery`; no devuelve instancia de escritura/lectura unificada.
2. **Scopes por alias** — Extender la API actual para que, además de `scope({ tableName })`, se pueda usar **`scope("nombreLogico")`** cuando la config declare el mapa alias → tabla (misma fuente de verdad que usa `initTracevault`).
3. **Menos trabajo manual** — El desarrollador deja de depender de aplicar a mano `generateInitSql` / SQL suelto para cada tabla nueva; el boot llama `await initTracevault(...)` y luego sigue usando las fábricas como hoy.

---

## 2. No objetivos (quedan fuera de 0.5.0 salvo decisión explícita)

- Sustituir o aliasar `emit` como `audit`.
- Un solo mega-objeto que combine pool + query + write retornado por `init`.
- Motor de migraciones tipo Knex/Prisma con versionado de esquema.
- `schema.tabla` calificado en SQL si no se extienden `assertValidTableName`, quoting en driver/reader y tests (si se desea, **tarea aparte** con su propio criterio de seguridad).
- API pública que exponga `tableName` en helpers tipo `scopes.resolve()` para consumo de aplicación (solo operadores / logs internos si acaso, no tipos exportados de uso general).

---

## 3. Contrato público propuesto

### 3.1 `initTracevault`

```ts
await initTracevault({
  driver: "postgres",
  connectionString: process.env.DATABASE_URL!,
  scopes: {
    default: { tableName: "audit_logs" },
    users: { tableName: "audit_user_events" },
  },
  // opcional, ver §5
  init?: { createTables?: boolean; createIndexes?: boolean },
});
```

- **Resolución:** `Promise<void>` (o `void` si se expone sync; preferible **async** por I/O).
- **Comportamiento:** validar opciones y nombres; si `createTables` (default `true`), ejecutar DDL idempotente **por cada** `tableName` único en `scopes` (misma semántica que `generateInitSql` hoy).
- **Conexión:** conexión **efímera** para el bootstrap: `pg.Client` o `Pool` con `max: 1`, cerrada al finalizar — no pool global propiedad de `init`.

### 3.2 Config compartida (fuente de verdad)

Nuevo tipo exportado (nombre tentativo):

```ts
export type TracevaultScopeTableConfig = {
  tableName: string;
};

export type TracevaultScopesMap = Record<string, TracevaultScopeTableConfig>;
```

- **`initTracevault`:** solo necesita `driver`, `connectionString`, `scopes` (y flags `init` opcionales). Recorre **todas** las entradas y asegura cada `tableName` (deduplicar por nombre físico antes de ejecutar DDL).
- **`createTracevault` / `createTracevaultQuery`:** cuando se usen aliases, además de `scopes` hace falta **`defaultScope: string`** que exista como clave en `scopes`, para saber en qué tabla escribe/lee la instancia raíz.

### 3.3 `createTracevault` (extensión)

- Opción A (recomendada): **nuevo overload o campo opcional** en `TracevaultConfig`:

  ```ts
  // además de los campos actuales:
  scopes?: TracevaultScopesMap;
  defaultScope?: string; // requiere scopes; root escribe en tableName del defaultScope
  ```

  - Si `scopes` + `defaultScope` están presentes: `tableName` raíz se deriva de `scopes[defaultScope].tableName` (o se exige que no haya conflicto con `tableName` explícito).
  - Nuevo método o sobrecarga: **`scopeByName(name: string)`** o extensión de **`scope()`** para aceptar `string | TracevaultScopeOverrides`:
    - `scope("users")` → resuelve `tableName` desde el mapa congelado en el factory (`Map`, O(1)).
    - `scope({ tableName: "x" })` → comportamiento actual (compatibilidad).

- Documentar precedencia si coexisten `tableName` root y `scopes` (recomendación: **error** si hay ambigüedad).

### 3.4 `createTracevaultQuery` (extensión)

- Misma forma opcional: `scopes`, `defaultScope`, y `scope("users")` / `scope({ tableName })` con las mismas reglas que el writer para mantener **simetría** lectura/escritura.

### 3.5 Exportaciones (`package.json` / `src/index.ts`)

- Exportar desde el entry principal:
  - `initTracevault`
  - Tipos nuevos relacionados (`InitTracevaultOptions`, mapas de scopes, etc.).
- `tracevault/query`: exportar extensiones de tipos y fábrica si aplica (o solo tipos en `index` si la query sigue en subpath — coherente con exports actuales).

### 3.6 `generateInitSql`

- **Mantener** como utilidad para quien migra con archivos SQL o herramientas externas.
- **Refactor interno:** extraer generación de DDL a una función compartida usada por `generateInitSql` **y** por el runner de `initTracevault`, para **una sola fuente de verdad** del esquema (evita divergencia).

## 4. Arquitectura interna

| Pieza | Responsabilidad |
|--------|------------------|
| `src/core/schema-ddl.ts` (nuevo, nombre tentativo) | Construir string o lista de sentencias SQL idempotentes para una tabla (extraído de `schema.ts`). |
| `src/core/schema.ts` | Reexportar o delegar en `schema-ddl` para no romper import `generateInitSql`. |
| `src/core/init-tracevault.ts` (nuevo) | Validar `InitTracevaultOptions`, abrir cliente/pool mínimo, ejecutar DDL por cada tabla, cerrar conexión. |
| `src/core/validator.ts` | `assertValidScopeName` (regex acordada); reutilizar `assertValidTableName`. |
| `src/core/tracevault.ts` | Resolver `scope(string)` con `Map` inmutable desde config; sin estado global. |
| `src/query/query.ts` | Paridad con writer para `scope(string)`. |
| `src/types/index.ts` | Tipos `TracevaultScopesMap`, extensión de `TracevaultConfig` / `TracevaultQueryConfig`. |

**Performance:** init = cold path, conexión corta; runtime = sin cambios en hot path salvo un `Map.get` al usar alias.

## 5. Opciones de `init` (defaults)

Propuesta de defaults conservadores para DX:

| Flag | Default | Notas |
|------|---------|--------|
| `init.createTables` | `true` | `CREATE TABLE IF NOT EXISTS` |
| `init.createIndexes` | `true` | Mismo bloque que hoy en `generateInitSql` |

Si en el futuro se añade “solo verificar sin crear”, sería un flag explícito `dryRun` / `verifyOnly` (fuera de alcance mínimo si no lo pedís).

---

## 6. Validación

- **Scope name:** p. ej. `^[a-zA-Z][a-zA-Z0-9_-]*$` (misma política en init y en factories).
- **Table name:** reglas actuales de `assertValidTableName` hasta que se soporte `schema.table`.
- **`defaultScope`:** debe existir en `scopes` cuando se usen aliases en factories.

---

## 7. Errores y compatibilidad

- Tabla existente con esquema **incompatible** con Tracevault: **fallar en init** con `ConfigError` / mensaje accionable (no intentar migración column-by-column compleja en 0.5.0).
- **Semver:** 0.5.0 — cambios **aditivos** si solo se añaden funciones y campos opcionales en config; breaking solo si se redefine comportamiento de `scope()` sin compatibilidad (evitar: sobrecarga `string | overrides` bien definida).

## 11. Riesgos / seguimiento

- **Índices / nombres:** tablas con nombres muy largos y truncado PG 63 chars (ya documentado en `generateInitSql`).
- **Duplicados:** dos aliases apuntando al mismo `tableName` — ejecutar DDL una vez por nombre físico único.
- **Futuro:** usuario solo lectura para consola; soporte `schema.table`; verificación estricta de esquema vía `information_schema`.
